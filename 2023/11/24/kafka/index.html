Kafka

MQ的使用的目的：

削峰 、 异步 、 解耦

本质：通信，消息的收发 ( 网络通信)

broker不对消息做处理

三个角色：producer broker comsumer

回忆过往使用关键词：

异步发送，ACK、leader再平衡、ZK、消费策略制定、消费者组中消费者上下线触发再平衡、多partition设计、动态扩缩容

发送细节：数据压缩 lz4、gzip、生产端：batch size linger.ms 设置

消费者批处理拉取消费

支持自定义序列化方式：默认json、kryo、protobuf

源码java部分设计：

零拷贝：生产端的mappedbytebuffer

多层级时间轮，相较于netty的时间轮改进，避免空推进浪费CPU

主从多线程版的Reactor（反应堆）通信模型

消费者组与协调者，单分区数据不会被同组多人消费

消费相关水位线（HW、LEO)，选主

什么是page cache

针对文件系统的缓存，将磁盘文件保留到内存当中，减少磁盘IO提高性能

Kafka大量使用了页缓存，消息都先被写入页缓存，有操作系统决定刷盘，但kafka也引入了同步刷盘和间断性强制刷盘，可以提高可靠性，但不建议这么做，将会大大影响kafka的性能，刷盘就该给os去操作，消息的可靠应当由kafka的消息确认机制和副本机制保证

kafka会将磁盘的一部分分区作为swap，把那些非活跃的进程调入swap分区当中，以保留出来内存给活跃的进程

被写入的page将会被标记为脏页，将会周期的刷盘 （ 超过某个门限 ，超过设置内存比例）

kafka follower如何与leader同步数据

数据只要被Leader写入log就被认为已经commit，这种情况下，如果leader挂掉，会丢失数据，kafka使用ISR的方式很好的均衡了确保数据不丢失以及吞吐率。Follower可以批量的从Leader复制数据，而且Leader充分利用磁盘顺序读以及send file(zero copy)机制，这样极大的提高复制性能，内部批量写磁盘，大幅减少了Follower与Leader的消息量差。

什么情况下一个 broker 会从 isr中踢出去

leader会维护一个与其基本保持同步的Replica列表，该列表称为ISR(in-sync Replica)，每个Partition都会有一个ISR，而且是由leader动态维护 ，如果一个follower比一个leader落后太多，或者超过一定时间未发起数据复制请求，则leader将其重ISR中移除 。

kafka 高性能

  • Cache Filesystem Cache PageCache缓存
  • 分区并行消费，充分利用多核
  • 顺序写 由于现代的操作系统提供了预读和写技术，磁盘的顺序写大多数情况下比随机写内存还要快。
  • Zero-copy 零拷技术减少拷贝次数
  • 文件读写刷盘上分区分段加索引
  • Batching of Messages 批量量处理。合并小的请求，然后以流的方式进行交互，直顶网络上限。
  • Pull 拉模式 使用拉模式进行消息的获取消费，与消费端处理能力相符。

kafka的message格式

一个Kafka的Message由一个固定长度的header和一个变长的消息体body组成

header部分由一个字节的magic(文件格式)和四个字节的CRC32(用于判断body消息体是否正常)构成。

比如是否压缩、压缩格式等等);如果magic的值为0，那么不存在attributes属性

body是由N个字节构成的一个消息体，包含了具体的key/value消息

数据结构：deque
