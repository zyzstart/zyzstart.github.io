Mysql的逻辑架构

连接层

客户端的连接服务、包含本地sock通信和大多数基于客户端/服务端类似实现TCP/IP的通信

主要的任务是：处理连接、授权认证、相关的安全方案

引入了线程池，为通过认证、安全接入的客户端提供线程处理任务。

同样在该层上可以实现基于 SSL 的安全链接。服务器也会为安全接入的每个客户端验证它所具有的操作权限。（用户和用户权限相关）

服务层：

核心服务功能，如SQL接口、完成缓存的查询、SQL的分析以及优化、部分的内置函数的执行

所有的存储引擎功能将在这一层实现、如函数、过程

服务器会解析查询并创建内部的解析树，完成对其相关的优化，如确定查询表的顺序，是否利用索引等，最后生成相应的执行操作。

如果是 select 语句，服务器还会查询内部的缓存。如果缓存空间足够大，这样在解决大量读操作的环境中能够很好的提升系统的性能。

引擎层

负责数据的存储和提取、存储引擎主要包括InnoDB、MyISAM（5.5之前）、Memory（缓存，了解部分）

InnoDB底层采用B+树数据结构存储索引

存储层

数据存储层，主要是将数据存储在运行于文件系统上，并完成与存储引擎的交互。

工作流程

①建立连接

MySQL 客户端通过协议与 MySQL 服务器建连接，发送查询语句。

②查询缓存

检查查询缓存中是否存在要查询的数据。

  • 命中：直接返回结果。

    不会再对查询进行解析、优化、以及执行。它仅仅将缓存中的结果返回给用户即可，这将大大提高系统的性能。

  • 未命中：解析 SQL 语句。

③语法解析器和预处理

MySQL 首先根据语法规则验证 SQL 语句的语法正确性，然后通过关键字将 SQL 语句进行解析，并生成一棵对应的『解析树』。预处理器则根据一些 MySQL 规则进一步检查解析树是否正确。

④生成执行计划

当解析树被确认为正确，接下来就会由查询优化器将解析树转化成执行计划（profile）。一条查询可以有很多种执行方式，最后都返回相同的结果。优化器的作用就是找到这其中最好的执行计划。

⑤按计划执行并获取数据

MySQL 默认使用 B+Tree 索引，并且一个大致方向是：无论怎么调整 SQL，至少在目前来说，MySQL 最多只用到表中的一个索引。

表空间

在 InnoDB 存储引擎下，表相关的所有数据（比如业务数据和索引数据）都储存在表空间（tablespace）中。每张表都有一个自己的文件（.ibd）去储存相关数据（开启独立表空间设置）。

表空间又可以细分为 segment，extent，page，row。

段

表空间又包含多个段（segment），常见的数据段有：

  • Leaf node segment 数据段，存储当前表中的数据
  • Non-Leaf node segment 索引段，存储当前表中的索引

区

段包含很多个区，每个区始终为 1MB 。区由多个连续连续的页组成，页的大小通常是 16KB，所以一个区可以有 64 （1024/16=64）个连续页。

页

页是 InnoDB 与磁盘交互的最小单位。从磁盘上读取数据，一次性是读取一页数据。将内存中的数据落盘到硬盘上，也是操作一页数据。

行

系统表空间

系统表空间是所有表共享的，它保存了数据表结构，事务信息 等

独立表空间

独立表空间是指每张表单独用一个文件储存每张表对应的数据和索引，文件扩展名为 ibd。

每个数据库会有一个对应目录用于保存当前数据库中的数据文件

索引

1、概念介绍

索引类型 索引特点
单列索引 即一个索引是只根据一个字段创建的，里面只包含单个列， 一个表可以有多个单值索引（也叫单列索引）
联合索引 即一个索引包含多个列（也叫多值索引、多列索引）
唯一索引 索引列的值必须唯一，但允许有空值，空值可以有多个
主键索引 设定为主键后数据库会自动建立索引，InnoDB 为聚簇索引

单列索引，根据某个字段建立的索引

联合索引，根据多个字段建立的索引，和查询顺序无关。需满足最左前缀原则，建立联合索引最左边的字段必须在查询字段中出现

带头大哥必须在，当索引中间某个字段不存在时，后面索引部分失效

唯一索引：基于唯一字段建立的索引

主键索引：基于主键创建的索引

表锁

​ 元数据锁

元数据锁的主要作用是在数据库中对元数据进行保护，防止并发操作导致元数据不一致或损坏。

​ 意向锁（相遇一个意向的标志）

通过使用意向锁，数据库系统可以更有效地管理锁资源，减少锁竞争和死锁的发生。当一个事务要在某个资源上获取锁时，它可以首先获取相应的意向锁，然后再尝试获取具体的锁。这样，其他事务可以根据意向锁来判断是否需要等待或者可以继续执行。这种机制可以减少事务在并发访问时的冲突，提高系统的吞吐量和性能。

需要注意的是，意向锁本身并不直接保护资源，而是用来表示事务对资源的意图，实际的锁是由具体的锁类型（如行级锁、表级锁）来保护资源的。因此，在使用意向锁时，必须结合具体的锁类型来使用，以确保对资源的访问是正确和安全的。
